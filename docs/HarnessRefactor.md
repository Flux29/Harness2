# Harness2 Refactor Plan

**Source:** github.com/Flux29/Harness @ `77f465e` · **Target:** new fork `Harness2`
**Driving document:** `HarnessCritique.md` (2026-07-03) and the alignment triage that followed it
**Ordering principle:** cheapest and least ambiguous first; anything requiring a contract decision last. Every phase ends with a gate: CI green + the phase's parity matrix verified. No phase begins until the previous gate passes.

**Triage legend used throughout** (from the alignment discussion):
- **[D→C]** docs move to match code (descriptive drift; code is the verified reality)
- **[C→D]** code moves to match docs (docs are a load-bearing contract, code leaked)
- **[NEW]** both move; the contract itself needs revision first, then code follows

## Phase 0 — Fork, baseline, and CI scaffold

Nothing is refactored in this phase. Its job is to make every later phase measurable.

**0.1 Create the fork — filtered first commit.** Fork `Harness` → `Harness2`, but the first commit is **not** the untouched tree: it is the tree minus the exclusion manifest (0.1a). Rationale: git history is permanent — anything present in commit 1 is in the fork's history forever, so known-contaminated artifacts must be excluded *before* the first commit, not deleted in Phase 1. From here forward, small commits with real messages — the single-squash publication pattern is retired. If the Windows mount still can't host git, the working copy lives in a normal directory and syncs to the mount, not the other way around; document this in the README's publication note (it's currently backwards).

**0.1a Exclusion manifest (`EXCLUSIONS.md`, committed in Harness2 for auditability).** Admission criteria are strict, and both must hold: **(a) high-certainty inert** — zero import, runtime, test, or tooling references anywhere in the tree, so exclusion cannot change any behavior; **(b) contamination or security risk** — machine state, personal data, raw prompts, or runtime output that should never enter a public fork's permanent history. Verified against the tree:

| Excluded | Category | Inertness evidence |
|---|---|---|
| `cache/pythoncore-3.13-hklm-before-agentic.reg` | Windows registry hive export — machine state | no reference anywhere in code, docs, or config |
| `projects/agent-web/body.json` | curl scratch payload | duplicated inline in HANDOFF step 3; nothing reads the file |
| `projects/agent-web/frontend/tsconfig.tsbuildinfo` | compiler incremental state | regenerated by `tsc`; gitignore gains `*.tsbuildinfo` |
| `vendor/pydantic-deepagents/jobs/terminal-bench/**` | benchmark runtime logs (env details, paths, exception dumps) | outside `pydantic_deep/`; `revendor_check.py` only compares `pydantic_deep/**/*.py`, so exclusion is invisible to vendor integrity |
| `docs/Deep_Agents.MD` | scratchpad containing raw prompts, a GitHub username directive, pasted console errors | referenced by nothing in README/PDR/ADRs; its still-relevant TODO items are triaged into Harness2 issues **before** exclusion (this absorbs Phase 1 step 1.4, which is deleted from Phase 1) |

**Deliberately NOT excluded**, despite being critique findings, because they fail criterion (a) or belong to a later decision: the legacy eval-optimizer modules (`graph.py`, `loop.py`, `validate.py`, dead schemas) are **test-load-bearing** — `test_smoke.py` imports `agents.Verdict` and `test_validate.py` imports `validate.parse_artifact` — and their fate is ADR 6.5's, not a commit-zero guess; files containing personal paths (`verify_core.py`, READMEs, HANDOFF) need *edits*, not exclusion — an edited line in history is harmless, it's whole-file contamination the manifest targets; and everything else under `vendor/pydantic-deepagents` stays byte-identical per the pristine-vendor rule.

**Baseline consequence:** because commit 1 is filtered, byte-parity between Harness2's first commit and Harness1 is replaced by the manifest plus a sha256 list of every retained file (`EXCLUSIONS.md` appendix). All Phase 0.2 baselines are captured **from the original Harness repo**, which continues to exist as the untouched reference — nothing in the parity harness ever depended on the excluded files.

**0.2 Capture the v1 baseline.** Before changing anything, record the observable behavior of Harness1 so parity is checked against evidence, not memory:

- Run `agent-web` tests, `eval-optimizer` tests, and `verify_core.py`; archive full output to `baseline/test-output-v1.txt`.
- Run one live smoke (`LIVE WIRE OK` flow) and one `TestModel` run per thread scenario; capture the **raw SSE transcript** to `baseline/sse-v1/`. These transcripts are the ground truth for the wire-protocol parity matrix. **All live inference — in baseline capture and in every later phase's verification runs — is pinned to `openrouter:z-ai/glm-5.2`** (the decided default, see 3.4); `TestModel` remains the substitute for offline/CI runs only. Mixing models between baseline and verification runs would make Matrix E diffs unattributable to code changes.
- **Capture Logfire telemetry for each live baseline run.** The Logfire MCP is already wired; for every live run above, record the trace id and export a span summary (span names, parent/child structure, model-call count, tool-execution count, MCP `tools/list` calls, token usage and cost fields, service name) to `baseline/traces-v1/<scenario>.json`. This is the ground truth for Parity Matrix E.
- Capture `GET /healthz` and `GET /debug/mcp` JSON to `baseline/endpoints-v1/` — the latter captured **twice**: once with the code-default `MCP_ENABLE` and once with the deployed roster (`context7,deepwiki,github,logfire`), so both configurations have a baseline (see the Matrix A roster note).
- Serialize one `history.json`, one `Settings()` instance (as dict), and one `HarnessForkReport` from a `TestModel` fork run to `baseline/schemas-v1/`.

**0.4 Build the parity harness.** A small pytest module (`parity/`) that loads each baseline artifact and asserts field-by-field equivalence against the current code, using the matrices defined below. The matrices are the **human-readable spec**; their disposition data is mirrored in the **machine-canonical manifest** (`parity/manifest.yml`, Gate 2 in 0.5), and the manifest — not the markdown — is what CI enforces. Rows marked **intentional change** are updated *in the same PR* that changes the behavior, with the ADR/step reference. Unexplained diffs fail CI.

**0.5 The gate system (adapted from the five-gates large-refactor pattern).** The pattern's core insight transfers directly even though its foundation gate doesn't: their refactor had a machine-checkable invariant ("function bodies never change, only locations do") and every gate existed so that Claude's job was to *satisfy a deterministic checker* rather than to *be trusted*. Harness2's invariant is different — "observable behavior identical except where explicitly declared" — but it is equally machine-checkable, and each of their five gates maps onto it. Each gate catches a different class of mistake; that separation of failure classes is the design, not an accident.

**Gate 1 — Equivalence (their AST-equality foundation, tiered for our case).** Their foundation was AST equality because *nothing* was allowed to change. Ours can't be — most phases change behavior deliberately — so equivalence is a three-tier ladder, and every PR declares which tier each touched file sits on:
- **Tier 1, byte equality (sha256):** the vendor tree and any file a PR claims not to touch. Already the `EXCLUSIONS.md` appendix mechanism; the gate extends it per-PR.
- **Tier 2, AST equality:** relocations and formatting-only changes — logic identical, bytes may differ. Implemented as a small `ast.parse`-and-normalize comparator (strip locations, docstring-insensitive option off by default). This is the exact article mechanism, scoped to the slices of Harness2 where the "bodies don't change" invariant genuinely holds: the 6.5 legacy relocation (if the ADR chooses `legacy/` over deletion, every moved function must AST-match its original), any ruff-format sweep, and the 1.5 personal-path scrubs in *code* files (comment/string edits must leave the AST's executable structure intact where the edit claims to be doc-only).
- **Tier 3, behavioral parity:** everything that legitimately changes — covered by Matrices A–E, keyed to a manifest entry.
The gate's rule: no file changes without a declared tier, and Tier 1/2 declarations are verified mechanically. A "hygiene" PR that fails AST equality is either mislabeled or contaminated with logic changes — exactly the copy-drift class the article built the gate for.

**Gate 2 — Manifest provenance (their YAML mapping, generalized from "moved functions" to "changed behaviors").** Their `old_path:new_path` YAML with two lints is the strongest transferable idea, because it fixes a real weakness in this plan as previously written: the matrices' disposition columns were enforced by review discipline — precisely the control that degrades under fatigue and that Claude can drift past. Harness2 adopts the structure wholesale, widened one level:

```yaml
# parity/manifest.yml — the canonical state of the refactor
findings:
  crit-4.3-thread-slug:
    status: changed          # identical | changed | removed
    step: "4.3"
    matrix_rows: ["D:workspace-layout"]
    tests: ["tests/test_deps_factory.py::test_slug_no_collisions"]
  crit-3.2-422-encoding:
    status: changed
    step: "3.2"
    matrix_rows: ["C:422-body"]
    tests: ["tests/test_e2e_agui.py::test_422_body_is_json_object"]
relocations:                  # populated only if ADR 6.5 chooses relocation
  eval_optimizer.validate.parse_artifact: eval_optimizer.legacy.validate.parse_artifact
```

And their two lints, verbatim in spirit: **lint one (completeness)** scans the manifest against the critique — every finding ID from `HarnessCritique.md` §8 must have an entry, and every `relocations` source must no longer exist at its old path while the target exists (guaranteeing the manifest is the canonical state of the migration, their exact phrase and exact mechanism); **lint two (test coverage)** verifies every `status: changed` entry names at least one test that exists and passes, and every relocation entry is covered by the Gate 1 AST comparator. Claude populates the manifest as it works; the lints make skipping an entry — or skipping a test — a CI failure rather than a review catch. This converts the standing rule "one finding, one PR, one matrix delta" from a norm into a machine veto.

**Gate 3 — Endpoint parity.** Already this plan's Matrices C and D (SSE event shapes, interrupt/resume payloads, `/healthz`, `/debug/mcp`, deployment-config row) plus Matrix E as the telemetry extension. The article's gate validates the approach at 130k-line scale; no structural change needed — the only amendment is bookkeeping: endpoint-parity failures must reference the manifest entry that predicted them, tying Gate 3 into Gate 2.

**Gate 4 — Duplication detection.** The article's gate catches the classic agent failure: Claude *copies* instead of *moves*, and both survive. Harness2 needs this gate twice over, because the v1 tree already has the disease organically — two fork-report vocabularies in `schema.py`, `features/` vs `toolsets/` coordinator copies in the vendor tree, four legacy strata. Mechanism: an AST-hash duplication scan over first-party code (function-level normalized hashes; jscpd or a ~50-line script), compared against `parity/dup-allowlist.yml`. The allowlist starts by *inventorying the known v1 duplicates* (dated, with the owning ADR noted — the fork-report pair is owned by 6.5), and the gate's rule is monotonic: **the allowlist may only shrink, never silently grow.** A new duplicate requires a dated, expiring allowlist entry in the same PR — which is exactly how the 5.1 parallel-run dual-write is handled (an intentional, temporary duplication with an expiry tied to its cutover criterion). ADR 6.5's exit criterion gains a number: first-party allowlist reaches zero (vendor-tree duplicates are outside our blast radius per the pristine rule and stay inventoried-but-exempt).

**Gate 5 — Import contract (their circular-import test, strengthened).** Cycle detection over first-party packages is the cheap adopt-outright piece (`import-lint` job; grimp or a topological check on the import graph). The stronger form Harness2 adds is a **layer contract**: declared rules like "nothing in the live path imports `legacy/`" (post-6.5), "`agent_web` never imports `eval_optimizer`", "first-party code imports `pydantic_deep` only through its public `__init__` surface — never `pydantic_deep.toolsets.forking` (the deprecated shim)". The critique's finding that the legacy strata are "interleaved and importable from everywhere" is precisely a missing layer contract; this gate is its permanent enforcement, and it also fails the build if anyone reintroduces the deprecated vendor import path.

**0.6 Execution scaffold (if Claude Code performs the refactor).** The agent's context is a budget; the vendor tree is ~86k of the repo's ~88k Python lines and is read-only by standing rule, so nothing in it should ever enter a working session's grep radius. Add a root `.ignore` scoping out `vendor/` (except `vendor/*.md`, `vendor/patches/`, and the two verify scripts, which the refactor legitimately reads), `uv.lock` files, `frontend/package-lock.json`, `baseline/`, and `workspaces/`. Add a per-project `CLAUDE.md` carrying exactly the scoped commands (`uv run pytest -q`, `uv run ruff check .`, `uv run pyright`) plus the three standing rules an agent must never rediscover mid-session: vendor is read-only, the manifest must be updated in the same change as the behavior, and PRs declare a Gate 1 tier per touched file. Batches follow the plan's existing granularity (one finding per PR ≈ the 5–20 file logical-unit guidance); Phase 1 doubles as the pilot batch that tunes prompts and gate friction before anything behavioral is attempted.

## Phase 4 — Correctness fixes with unambiguous contracts

These change behavior, but the *correct* behavior is not in dispute — no design decision needed, just care.

4.0 **Coverage-first for the fork engine (prerequisite for 4.5, 4.6, 5.2, and 6.1).** The generic large-refactor rule — if a critical area lacks tests, add them *before* migrating it — applied to the one region of Harness2 where it bites: `forking.py` has essentially zero behavioral coverage, and four later steps reshape it. Before any of them, land `TestModel`-based tests pinning **current** v1 behavior: the fallback path fires on a raised exception in deterministic selection; the `>0` threshold merges a partial-pass branch; the timeout path returns with branches unresolved; `save_winner_dir` copies on the fallback path (the very bugs 4.6 will fix get pinned red-side first, per the red-test-first rule). Matrix B currently verifies the *serialization* of a component nothing exercises; 4.0 gives it behavior to stand on, and gives Gate 2's lint two something real to point at when those manifest entries flip to `changed`.

4.1 **Decouple `Settings.from_env()` from NVIDIA [C→D, ADR-0003].** Provider credentials are validated lazily per resolved model family: `build_model()` raises a clear error naming the missing key for the provider actually selected (including the currently-silent empty OpenRouter key). Tests: OpenRouter-only env constructs Settings and builds an OpenRouter model; NVIDIA-only likewise; missing key for the *selected* provider raises with the provider named.
4.2 **Close the retry-coverage gaps** the docstring claims: `memory_pg`'s embedding client uses the shared retrying transport; the `provider:model` pass-through branch either gains the transport or the docstring's claim is narrowed to match (state which in the PR — either resolution is acceptable, the lie is not).
4.3 **Fix `thread_slug` collisions:** sanitized-name plus short content hash of the original id (e.g. `slug[:48] + "-" + sha256(id)[:12]`). This is a workspace-relocating change for *colliding* ids only; Matrix D's rule — previously-non-colliding ids map to identical directories — is the acceptance test, verified by hashing the baseline workspace layout. Existing-workspace migration note goes in the changelog (Harness1 threads with exotic ids won't resume their old workspaces; acceptable for a fork).
4.4 **Serialize runs per thread:** an in-process `asyncio.Lock` per thread id around load→run→save in `POST /agent`. Concurrent requests on one thread queue; distinct threads unaffected. New E2E test: two simultaneous posts to one thread, assert both messages present in final history (this test fails on v1 — include the v1-red run in the PR description as evidence).
4.5 **Instrument fork selection provenance:** `HarnessForkReport` gains `selection_path: Literal["deterministic","judge_fallback"]` and the fallback logs the swallowed exception. 4.6 **Fix winner-materialization correctness:** `save_winner_dir` copies only when the merge path actually flushed a winner (deterministic pick succeeded), never on the fallback/abort paths; `_wait_for_branches` timeout explicitly cancels or fails outstanding branches before selection rather than selecting over live branches. 4.7 **Rebuild-safe MCP/static wiring:** either the agent's toolsets are rebuilt when a registry entry transitions to ready, or `/debug/mcp` reports the *agent's startup snapshot* alongside live registry status so the two can't silently disagree; same principle for the static mount (`/healthz`'s `frontend_built` and the actual mount must not contradict — mount lazily or report honestly). Minimal-change option is the honest-reporting one; pick per taste, the contract is "no endpoint may report a state the agent doesn't have."

**Exit gate 4:** CI green; parity matrices pass with documented flips; the two new E2E tests (concurrency, fork-timeout) green; a full `TestModel` fork run's report field-compares against the Matrix B baseline.

## Phase 5 — Trust-boundary repairs [C→D] (the contract is right; the code leaked)

Higher blast radius than Phase 4, but still not ambiguous — ADR-0012's trust model and the approval-gating principle are contracts you've affirmed. Sequenced after Phase 4 because 4.4's locking and 4.3's slugging are prerequisites.

5.1 **Move authoritative history outside the agent-writable root — via parallel-run, not cutover.** `history.json` relocates to a server-only directory (e.g. `workspaces/_history/<slug>.json` or a sibling `state/` tree) that is *not* inside any `LocalBackend` root. Adopting the parallel-run gate rather than a hard cutover: for a verification window, the server **dual-writes** history to both the old and new locations and diffs them on every save (any divergence logs loudly and fails the live parity run); **reads cut over to the new location only after N clean sessions** (pick N; the exit gate uses it). The dual-write is registered as a dated, expiring entry in `parity/dup-allowlist.yml` (Gate 4) so the intentional duplication cannot silently become permanent — its expiry *is* the cutover deadline. This converts the riskiest persistence change from "trust the tests" to "observed equivalence," the same epistemics as Matrix E. Tests: an agent file-tool `ls`/`read` of the workspace cannot see history (either copy); the E2E history-persistence test passes against the new path. Matrix D location row flips; message schema row stays identical. Migration: a one-shot relocation of existing files, or accept fresh histories on the fork — state which in the PR.
5.2 **Close the fork-execution gate gap.** The contract: LLM-generated code does not execute on the host without either explicit approval or explicit configuration. Concretely: agent-web's `forking=` becomes env-gated like its siblings (`FORKING=0` default off, or on-with-interrupt — the *mechanism* follows what the harness supports for gating `test_command`; if the harness offers no interrupt hook there, the honest gate is the env flag plus a README warning symmetrical to the browser one). The asymmetry — shell gated, pytest-on-host ungated — is the defect; any resolution that removes it and is documented satisfies this step. eval-optimizer's headless path keeps host execution but the docstring disclosure is promoted to a required env acknowledgment (`EVALOPT_ALLOW_HOST_EXEC=1`) so "a machine you trust" is an explicit configuration, not a docstring.
5.3 **Pin the bind address as a stated invariant:** `--host 127.0.0.1` explicit in `Start-AgentWeb.ps1`, `Register-StartupTask.ps1`, and the RUNBOOK smoke doc. One line each; it converts the implicit mitigating factor from the critique into a declared property, ahead of the Phase 6 posture ADR.

**Exit gate 5:** CI green; new negative tests (agent cannot read history; fork disabled by default produces no fork toolset; approval still required for execute) green; interrupt E2E suite unchanged and green — the approval flow is the thing most at risk of collateral damage here, and Matrix C is the tripwire.

## Phase 6 — Contract revisions [NEW] (decide first, then implement)

Everything here needs a decision recorded as an ADR *before* code changes. Each gets an issue (from 1.4) and an ADR number in Harness2's sequence. Ordered roughly by how much downstream code hangs on the answer. These are deliberately last: they're the items where "align code to docs" and "align docs to code" both canonize something wrong.

**6.1 — ADR: fork selection metric (replaces the self-graded ratio).** The open question is what signal ranks branches when branches author their own tests. Candidate directions to weigh in the ADR (not prescribed here): a fixed task-supplied test suite injected into every branch; a shared held-out suite the branches never see; ratio plus a test-quality floor (minimum assertion count / mutation-lite check); or ratio for filtering plus judge for ranking among passers. The ADR must also resolve 6.1a as a sub-decision: **the merge threshold** (currently `>0` merges a 1-of-10 branch as "winner") — define `any_viable` numerically. Code and Matrix B semantics follow the ADR.

**6.2 — ADR: fallback policy for selection failures.** With 4.5's provenance data in hand: does an exception in deterministic selection fall back to the judge (current), abort the fork (fail-loud), or retry deterministic once then abort? The bare-except-to-judge path silently violates ADR-0011's own deterministic mandate; whatever replaces it, the choice of failure mode is yours and the implementation is small once chosen.

**6.3 — ADR: checkpoint persistence scope.** The honest fork of the question from the alignment discussion: is cross-request rewind a real requirement? **Option A:** it is — implement a durable per-thread store (file-backed next to history in the 5.1 server-only tree is the minimal version) and keep ADR-0015's claim. **Option B:** it isn't — downgrade the documented claim to per-run checkpoints and keep `InMemoryCheckpointStore`. Either resolves the critique's top congruency finding; only you know which matches actual usage. The parity impact differs (A adds files to Matrix D; B changes only prose), so the ADR lands before any code.

**6.4 — ADR: composed security posture.** The critique's point was that every slice was individually approved and nobody assessed the assembly. One ADR that states the threat model for an always-on local agent service: bind address (now pinned by 5.3), auth on `/agent` (none / token / OS-level), CORS scope, which execution surfaces exist and how each is gated (execute, forking post-5.2, browser), and what the Task Scheduler auto-start implies. This ADR also unblocks the deferred CD decision from 0.3 (auto-deploy is only sane once the posture is written down).

**6.5 — ADR: disposition of the legacy strata in eval-optimizer.** Four superseded generations (`graph.py`, `loop.py`, `validate.py`/`validate_check.py`, the seven dead schema models, the dead-schema paths in `agents.py`). Options: delete (git history now preserves them — that's what Phase 0.1 bought), move to `legacy/` outside the package's import path, or extract the genuinely-reusable pieces (`parse_artifact` has tests and a plausible future) and delete the rest. The ADR should also rule on the two parallel fork-report vocabularies and on `memory_pg`'s seven pre-created agent schemas (keep as ADR-0004 infrastructure, or trim to the roster the post-pivot roadmap actually names). This is last-of-the-code-items because deletion is trivially easy and choosing *what* is dead requires you to affirm the roadmap. **Gate-backed exit criteria, whichever option wins:** if relocation — every moved function passes Gate 1 AST equality against its original and every move is a `relocations` entry in the manifest (Gate 2 lint one verifies old-path-gone/new-path-present); the first-party `dup-allowlist.yml` reaches **zero** (Gate 4 — the fork-report pair and any other inventoried duplicate must be resolved by this ADR, not carried); and the Gate 5 layer contract gains and enforces its post-6.5 rule ("live path never imports legacy"), making the containment the critique found missing a permanent CI property rather than a one-time cleanup.

**6.6 — Workspace coherence sweep (small ADR or PDR amendment):** one Python floor policy across the three projects (or a stated reason for divergence); the fate of the aspirational scaffolding layer (`catalogs/scaffolds.yml`'s undefined `${AGENTIC_SCAFFOLDS}`, the tool catalog nothing uses, `evals/agentic-smoke`) — adopt, trim, or delete; and whether `rules/hooks`' pip guard is hardened (proper tokenization instead of substring hints) or re-described as advisory. Low stakes individually; batched so the meta-workspace concept gets one deliberate yes/no instead of ambient decay.

## Standing rules for the whole refactor

**One finding, one PR, one manifest entry.** Formerly a review norm ("one matrix delta"), now a machine veto: Gate 2's lints fail CI if a PR changes behavior without its manifest entry, or flips an entry to `changed` without a named, passing test. A PR that flips a parity row it didn't declare fails the build, not just review.
**Every touched file declares its equivalence tier** (Gate 1): byte-identical, AST-identical, or behavioral-parity-with-manifest-entry. Hygiene and relocation PRs are verified mechanically at Tier 1/2 — Claude's job is to satisfy the checker, not to be trusted.
**Red-test-first for defects** (3.6, 4.4 pattern) **and coverage-first for under-tested regions** (4.0): pin the bug — or the region's current behavior — with tests on v1 before changing it, so the change's evidence is in the diff.
**Intentional duplication expires** (Gate 4): the dup-allowlist only shrinks; any new entry carries a date and an expiry tied to a cutover criterion (5.1's dual-write is the template).
**No safety claim is ever downgraded to match code** — the Phase 5/6 boundary exists precisely to prevent that shortcut under schedule pressure.
**Docs regenerate at phase ends, not continuously:** PDR and README get a sync pass at gates 2, 5, and 6; ADRs are written when decisions are made, never retroactively.
**The vendor tree stays pristine throughout** — nothing in this plan touches `vendor/pydantic-deepagents` beyond the runtime-log exclusion in 0.1a; if any phase seems to require a vendor change, that's a new patch file plus a `VENDOR.txt` entry, not an edit. Gate 5's layer contract additionally pins the *import* surface: first-party code reaches the harness only through `pydantic_deep`'s public interface.
