# Harness Repository Critique

**Repository:** github.com/Flux29/Harness · **Reviewed:** 2026-07-03 · **HEAD:** `77f465e` (single commit)

**Scope:** Critique only — no fixes, no repair plan. Documentation (README, PDR, HANDOFF, ADRs) was used as a map of *claims*; every finding below is grounded in the actual files. First-party code reviewed in full (`projects/agent-web`, `projects/eval-optimizer`, `catalogs/`, `rules/`, `infra/`, `evals/`, `vendor/` meta layer); the vendored upstream (`vendor/pydantic-deepagents`, ~88k lines) was reviewed structurally and at the patch site, not line-by-line.

---

## 1. Executive assessment

The repository is two projects in visibly different phases wearing one architecture. `agent-web` is a small, coherent, well-tested web service whose real design decisions (server-owned history, per-thread isolation, resilient MCP wiring, approval interrupts) are genuinely implemented and genuinely tested. `eval-optimizer` is an archaeological dig: one live path (`forking.py`/`fork_check.py`) buried under four superseded generations of itself, all still importable, all still shipping.

The documentation is unusually good at recording decisions and unusually bad at staying synchronized with the code. Several load-bearing claims in the PDR and ADRs are contradicted by the implementation — most seriously around checkpointing, the history trust model, and the approval-gating story. The repo's most consequential weaknesses are not sloppy code; they are places where two correct-looking components silently disagree about a contract.

## 2. Repository-level completeness

**The entire history is one squashed commit.** `git log` shows a single commit ("AgenticWork: deep-agent harness + AG-UI frontend (ADR 0001-0016)"). The PDR explains why (git cannot run inside the working mount; the tree was copied out and pushed). The consequence is that every evidentiary claim in the docs — "verified 2026-07-02," "10/10 tests green," "trace 019f215d," "ADR resolution logs carry the receipts" — is unverifiable from the repository itself. The ADR resolution logs referenced repeatedly by PDR.md and HANDOFF.md as the proof layer are not reachable from a clone: the Logfire traces live in a private project and the incremental history that would corroborate the timeline was destroyed by the copy-and-init publication path. A repo whose stated epistemology is "telemetry is the impartial witness" ships with none of the witnesses.

**`Obsolete/` is referenced as if it exists, but it is gitignored and absent.** README's layout table lists `Obsolete/` as a directory; `docs/adr/README.md` says the pre-relocation ADR originals are "parked in `Obsolete/eval-optimizer-docs-adr/`"; VENDOR.txt says the removed vendor `.git` and `.venv` are "parked in `Obsolete/vendor-pydantic-deepagents/`". None of that content is in the repository. For the vendor case this creates a real provenance gap: VENDOR.txt pins upstream to commit `f2224c5`, but the artifact that could prove it (the dead `.git`) was parked into a directory a clone never receives. `revendor_check.py` can verify a *future* re-vendor against a fresh clone; nothing in the repo verifies the *current* vendored tree.

**No root LICENSE.** The vendored upstream carries its MIT license, but the first-party code (`projects/`, `catalogs/`, `rules/`, `infra/`) has no license file at all. For a public GitHub repository this leaves the legal status of everything outside `vendor/` undefined.

**No CI for first-party code.** The only workflows in the tree are the upstream's (`vendor/pydantic-deepagents/.github/workflows/`), which do not run in this repo and would not test this repo's code if they did. The 13 agent-web tests, the eval-optimizer smoke tests, `verify_core.py`, and `revendor_check.py` all exist — and nothing runs any of them automatically. Given that the stated verification discipline is the repo's proudest feature, the absence of even a minimal test workflow is a notable incompleteness.

**Stray artifacts are committed.** `cache/pythoncore-3.13-hklm-before-agentic.reg` is a Windows registry hive export — a machine-state backup with no relationship to the codebase. `projects/agent-web/body.json` is a leftover curl scratch payload (it duplicates the inline JSON in HANDOFF step 3). `projects/agent-web/frontend/tsconfig.tsbuildinfo` is a compiler incremental-state artifact; `.gitignore` excludes `dist/` and `node_modules/` but missed it. Inside the vendored tree, `jobs/terminal-bench/` carries seven timestamped benchmark-run directories of logs, exceptions, and reward files (~100 files) — runtime output, not source, and VENDOR.txt's own note about excluding runtime dirs was not applied to it.

**`docs/Deep_Agents.MD` is an uncurated scratchpad committed as documentation.** It contains a raw TODO list, a pasted prompt ("Please create a repo in my Github space at https://github.com/Flux29/ named 'Harness'…"), pasted Vite build output, and a pasted PowerShell error fragment from `Start-AgentWeb.ps1` — all interleaved with no structure. Its content also partially contradicts the ADRs (it asks to port "all functionality from the TUI AND the Deep Research app" into the web frontend, while ADR-0015 explicitly rules the `apps/` out of scope). As the only file in `docs/` besides the ADRs and the startup plan, it materially lowers the signal quality of the documentation layer.

**Personal environment details are baked into the tree.** `C:\Users\pollm\...` appears verbatim in README-adjacent docs, HANDOFF.md, eval-optimizer's README, and `verify_core.py`'s docstring. `docker-compose.yml` references `C:\Work\Repositories\RAG\infra\compose` by absolute path. `schema.py`'s module docstring cites `E:\Schema\Schema.txt` — a design document on a local drive that no reader of this repository can consult, referenced as the authority for the pipeline's stage flow.

## 3. Documentation ↔ implementation conflicts

These are cases where a written contract and the code disagree. They are the most serious category because both sides look correct in isolation.

**Checkpointing is advertised as persistent and is actually per-request ephemeral.** ADR-0015's table says checkpointing gives a "resume/rewind control in UI" and is "required by forking"; PDR lists `include_checkpoints=True` among the live features; `deps.py`'s docstring says each thread gets "its own checkpoint store." But `make_deps()` constructs a fresh `InMemoryCheckpointStore()` on **every POST /agent**. Any checkpoint the agent creates in run N is gone when run N+1 arrives — cross-request rewind and `fork:<id>` anchoring on prior checkpoints cannot work. `test_checkpoints.py` quietly sidesteps this: it saves and reads within a single `make_deps()` call, testing exactly the scenario that works and not the cross-request persistence the docs promise. The test's isolation assertion ("thread B's store has no checkpoints") is trivially true for the wrong reason — *every* store is empty at birth.

**The server-owned history trust model is undermined by where history is stored.** ADR-0012 is explicit: client-sent history is untrusted; the server-side store is authoritative. But `history._path()` writes `history.json` into `workspaces/<slug>/` — the exact directory that `make_deps()` hands to the agent as its `LocalBackend` root. The agent's own filesystem tools can read, edit, or delete the authoritative history of its own thread. Any prompt-injected content that convinces the model to write to `history.json` rewrites the server's source of truth, including pending-interrupt state. The trust boundary the ADR draws between client and server exists; the boundary between the *agent* and the server's record does not.

**The approval-gating story is internally inconsistent.** `settings.py`/`agent.py` gate the `execute` shell tool behind an AG-UI interrupt (`interrupt_on={"execute": True}`), and README warns that browser automation is not gated. But `build_agent()` unconditionally enables `forking=LiveForkCapability(test_command="pytest -q", max_branches=4)` — and per ADR-0011's own trade-off 1, that `test_command` runs LLM-generated code **on the host**, ungated, with no interrupt, regardless of the EXECUTE flag being 0. The repo's security posture is thus: typing a shell command requires human approval; writing arbitrary Python into a branch and having the harness execute it via pytest does not. The asymmetry is nowhere acknowledged — ADR-0011 discusses host execution for the *headless eval-optimizer* path, while agent-web inherited the same exposure silently, in an always-on service registered to start at every logon.

**"Provider-flexible" is contradicted by a hard NVIDIA dependency.** ADR-0003 establishes provider-flexible, per-role inference; `.env.example` configures every role via `openrouter:`; PDR describes `build_model()` resolving four provider families. Yet `Settings.from_env()` raises `RuntimeError` if `NVIDIA_API_KEY` is unset — even for a run that never touches NVIDIA. The fork path (`forking.py` → `Settings.from_env()`) cannot start on an OpenRouter-only machine. Meanwhile the key that *is* needed, `openrouter_api_key`, defaults silently to `""` with no validation, deferring the failure to the first HTTP call. The mandatory check guards the optional provider and the optional check guards the mandatory one.

**Stale model/provider facts persist across four documents.** `eval-optimizer/pyproject.toml`'s description still reads "pydantic-deepagents + NVIDIA GLM 5.1." The README's header says reasoning is "NVIDIA-hosted GLM 5.1" and its setup instructs pasting `NVIDIA_API_KEY` into `.env`. AGENTS.md's hard rules state "GLM 5.1 (NVIDIA) reasons" and "Read from `.env`." All of this predates the OpenRouter/GLM-5.2 migration and the secrets-in-USER-env-vars policy that `.env.example`'s own header now mandates ("SECRETS DO NOT GO HERE"). AGENTS.md is especially problematic because, per its own preamble, it is auto-injected into every agent and subagent — the harness is actively feeding its agents instructions that contradict the repository's current policy.

**Config template and code defaults disagree.** agent-web's `.env.example` sets `AGENT_MODEL=openrouter:z-ai/glm-5.2` while `settings.py`'s default is `openrouter:z-ai/glm-4.6` and `verify_core.py` smoke-builds `z-ai/glm-4.6`. eval-optimizer's `.env.example` sets `GLM_MODEL=z-ai/glm-5.2` while `config.py` defaults to `z-ai/glm-5.1` (and `test_settings_defaults` pins the 5.1 default). None of these is individually harmful; collectively they mean the answer to "which model runs by default?" depends on which file you read.

**PDR's test claim overstates the test suite's nature.** PDR describes `projects/agent-web/tests/` as "13 E2E tests." There are 13 test functions, but `test_deps_factory.py` (3) and `test_checkpoints.py` (1) are unit tests, and `test_agent_build.py` and `test_mcp_wiring.py` exercise construction, not end-to-end flows. Roughly seven are E2E. Minor, but symptomatic of docs written to impress rather than to describe.

**HANDOFF's live-smoke instructions contradict its own header.** The header records the final state as "single server on :8801 — 8000 held by Docker," while step 3 below instructs `uvicorn ... --port 8000`. The document declares itself historical, but it is also the only place the live-smoke procedure is written down, and the procedure as written collides with the documented port conflict.

**The agent's system instructions hard-code an MCP roster the config may not provide.** `agent.py`'s instructions name `github_*`, `context7_*`, `logfire_*` prefixes and forbid `gh`/`winget`. But the roster is env-driven (`MCP_ENABLE`, default `context7,deepwiki`): with defaults, the model is told GitHub work "MUST use github_* tools" that do not exist in its toolset, while simultaneously being forbidden the alternatives. Instruction content and toolset content are configured in two uncoordinated places.

**README's tracing claim overstates the default.** "Everything traced end-to-end with Logfire" — tracing is a no-op without `LOGFIRE_TOKEN` (correctly so, per `observability.py`), so the accurate claim is "traceable," not "traced." Related: `mcp.json` wires a `postgres` MCP server via `${DATABASE_URL}`, but agent-web's `.env.example` never mentions `DATABASE_URL`, and HANDOFF states "Docker/pgvector NOT required by agent-web." The postgres entry is configured, undocumented, and unusable as shipped.

## 4. Design weaknesses and internal conflicts in the live code

**The fork-selection metric is self-graded.** `BUILDER_INSTRUCTIONS` requires each branch to write "all source files AND a pytest test suite"; selection is by `test_pass_ratio` on those same tests. Each branch authors the exam it is graded on. A branch that writes three trivial assertions scores 1.0 and beats a branch with a rigorous suite at 0.9 — the metric rewards weak tests. This is a direct Goodhart exposure, and it clashes with the repo's own stated principle (AGENTS.md: "The Optimizer never grades its own work"). ADR-0011 chose deterministic selection over the LLM judge to preserve ADR-0005's ranking discipline, but the determinism is anchored to a self-authored signal, which is arguably weaker than the judge it replaced. The shared task prompt does not fix the test suite; nothing normalizes test quality or count across branches.

**A partially-failing branch is merged as "the winner."** `forking.py` merges any branch with `test_pass_ratio > 0`. A branch passing 1 of 10 tests is merged, losers discarded, and `any_viable=True` is reported — `fork_check.py` then prints "C5 PASSED: a branch passed its tests." The schema comment defines `any_viable` as "a branch's tests passed," which the threshold does not enforce.

**The judge fallback swallows all exceptions.** The deterministic-selection block is wrapped in a bare `except Exception:` that falls through to `coordinator.resolve(strategy=MergeStrategy(kind="auto"))`. The comment frames this as an API-drift safety net ("if the API differs"), but it equally catches genuine bugs in the selection code, `HarnessBranchResult` validation errors, or a broken vendor patch — silently switching from the ADR-mandated deterministic path to the LLM judge the ADR explicitly rejected, with no log, no flag on the report, and no way to tell from `HarnessForkReport` which path chose the winner.

**Timeout handling leaves branches running into selection.** `_wait_for_branches()` polls for up to 900s and then simply returns the current statuses. Nothing cancels still-running branches; `branch_outcomes()`/`merge_or_select()` proceed against a coordinator with live branches. The `keep_artifacts=False` comment says "winner flushes to parent," and `save_winner_dir` then copies the *entire parent work dir* — if selection went through the fallback path, or a merge did not actually flush, the "winner files" copied may be the parent's plan-stage tree, still reported with `winner_dir` set and `any_viable=True`.

**Concurrent runs on one thread corrupt history.** `POST /agent` has no per-thread locking. Two simultaneous runs on the same `threadId` both `history.load()`, both run, and both `history.save(result.all_messages())` — last write wins, silently discarding the other run's messages (including any interrupt bookkeeping). The frontend's double-click guard on approvals shows awareness of the class of problem client-side; the server has no equivalent.

**`thread_slug()` weakens the isolation it implements.** Sanitizing to `[A-Za-z0-9_-]` and truncating at 64 characters means distinct thread ids collide: `a/b` and `a_b` share a workspace and a history file; any two ids identical in their first 64 sanitized characters do too. Since the slug is the *only* isolation key (backend root, history, checkpoints), a collision is a full cross-thread bleed. `test_slug_sanitizes` asserts the truncation as correct behavior rather than flagging the collision surface.

**The 422 error body is double-encoded.** `Response(content=json.dumps(e.json()), ...)` — `e.json()` already returns a JSON string, so the response body is a JSON *string literal* containing escaped JSON, declared as `application/json`. Clients parsing the error get a string, not the error object.

**Settings truthiness is inconsistent and case-sensitive.** `web_tools`/`tracing` use `not in ("0","false","no")` — empty string means **on**. The six deferred flags use `not in ("0","false","no","")` — empty string means **off**. And none lowercase the value first: `TEAMS=False` or `EXECUTE=FALSE` (natural on Windows/PowerShell, the target platform) evaluates truthy and *enables* the feature. For EXECUTE specifically, a user writing `EXECUTE=False` to disable shell execution turns it on. The test helper's comment ("BROWSER is a standard OS env var and burned us once already") records that ambient-env parsing already caused one incident; the parsing that caused it was kept.

**A comment is attached to the wrong flag.** In `settings.py`, `improve`'s trailing comment reads "search-on-demand tools instead of flat schemas" — that describes `tool_search`; the `improve` description sits on the line above. Trivial, but it is exactly the kind of drift the file's flag-per-line format was supposed to prevent.

**The agent and its MCP toolsets are frozen at startup.** `lifespan` builds the registry and the agent once. A server that was not ready at boot (missing token, cold `uvx` cache) never joins the toolset even if it becomes ready; `/debug/mcp` reports the registry's *current* status while the agent runs with the *startup* snapshot — the debug endpoint can honestly say "ready" for a server the agent cannot see. Similarly, the frontend static mount is decided once at `create_app()` time: if `dist/` did not exist at startup, building it later is reported by `/healthz` ("frontend_built") but never served until restart.

**Fork configuration is scattered and mutually inconsistent.** `max_branches` is 8 in eval-optimizer's builder (for 3 default approaches), 4 in agent-web, 4 in `verify_core.py`. eval-optimizer sets per-branch and aggregate budgets; agent-web's always-on forking sets no branch budgets at all (only the optional global `cost_budget_usd`), despite ADR-0011's budgets existing specifically so "a runaway branch can never repeat the 22-min/48-call burn."

**The retry-transport claim has coverage gaps.** `models.py` claims the shared retrying HTTP client "covers every call incl. embeddings." It covers Ollama-, OpenRouter-, and NVIDIA-built models. It does not cover the `provider:model` pass-through branch (returns the bare string; pydantic-ai builds its own client), and it does not cover embeddings at all — `memory_pg.py` constructs its own plain `OpenAI` client with no retry transport. The docstring describes an intention, not the implementation.

## 5. Dead weight, duplication, and scope residue

**eval-optimizer ships four generations of superseded architecture as live code.** PDR's "Legacy / superseded" table is honest about it: `graph.py` (240 lines, all node logic stubbed with `TODO(C)` markers and deterministic fake critics), `loop.py`, `validate.py` + `validate_check.py` (the Docker sandbox path the fork engine abandoned), and seven schema models (`ExecutionPlan`, `Candidate`, `Critique`, `RankedCandidate`, `ForkReport`, `BranchResult`, `ValidationResult`) superseded by the two `Harness*` models. The retained code is not quarantined — `agents.py` still builds planner/generator/evaluator agents against the dead schemas, `schema.py` interleaves live and dead models in one file, and the smoke tests exercise the dead `Verdict`/`parse_artifact` paths. Given that the workspace's own prior post-mortem (the earlier RAG repo) identified scope sprawl as the failure mode, and given the stated policy of parking dead files in `Obsolete/`, keeping four superseded layers importable inside the live package is a policy violation by the policy's own author. The retention rationale ("retained on disk, not the active path") also means every future reader — human or agent — must consult the PDR table to know which 60% of the package to ignore.

**Two parallel fork-report vocabularies coexist.** `ForkReport`/`BranchResult` (dead) and `HarnessForkReport`/`HarnessBranchResult` (live) express the same concept with different fields in the same module. `memory_pg.py` and its schema (`infra/initdb/01_init.sql`) pre-create schemas for seven agents ("the five general-purpose agents" of a Phase 4 that the AG-UI pivot arguably mooted) — infrastructure for a roadmap the ADRs no longer describe.

**The scaffolding layer is aspirational and disconnected.** `catalogs/scaffolds.yml` roots at `${AGENTIC_SCAFFOLDS}`, an environment variable defined nowhere in the repository. `catalogs/tools.yml` blesses nicegui/streamlit/gradio/marimo/playwright — none of which appear in any project. `evals/agentic-smoke` tests that numpy means numbers and scipy's normal CDF is 0.5 at zero; it validates the *catalog policy*, not the harness, and nothing references it. These files describe an "AgenticWork" meta-workspace concept that the two actual projects do not use.

**Python version floors conflict across the workspace.** `evals/agentic-smoke` requires `>=3.13,<3.15`; `eval-optimizer` requires `>=3.11,<3.14`; `agent-web` requires `>=3.10`; the vendored harness requires `>=3.10`; `catalogs/models.yml` declares 3.13 the compatibility default. There is no single Python that the catalog calls default, that evals accepts, and that eval-optimizer's upper bound permits at its ceiling — and HANDOFF records agent-web being verified on 3.10 while its sibling project could not sync at all. Three uncoordinated floors in one uv workspace is friction the consolidation pass did not consolidate.

**The vendored tree contains upstream's own duplication, and the patch touches only one copy.** `pydantic_deep/toolsets/forking/` is a deprecated shim (`__init__.py` re-exports from `features/forking`) — but the directory also still contains full stale copies of `coordinator.py`, `judge.py`, etc., that differ from the `features/` versions. The `0001-branch-outcomes` patch was applied only to `features/forking/coordinator.py`. Because the shim re-exports from `features/`, imports resolve to the patched code — but the tree now holds two coordinators, one patched and one not, and `revendor_check.py`'s "expected diffs match patch targets" discipline has to reason about which copy is which. This is largely upstream's mess, faithfully vendored; it still lives in this repo's blast radius.

**The rules layer enforces less than it claims.** `rules/hooks/agentic_pre_tool_use.py` blocks `pip install` unless the command contains any of five allow-hint substrings — including bare `.venv` or `conda` *anywhere in the string*. `pip install requests # .venv` passes. The hook is a reasonable nudge and an unreasonable thing to call "Enforced by the AgenticWork pre-tool-use policy," as AGENTS.md does.

## 6. Security posture

The individual exposures were mostly conscious trade-offs; the problem is their *composition*. Taken together, the shipped configuration is: an unauthenticated HTTP endpoint (`POST /agent`, no auth, CORS with wildcard methods/headers), auto-started at every logon via Task Scheduler with self-restart, running an agent that (a) always has host-executing fork tests enabled with no approval gate, (b) stores its own authoritative history inside its writable workspace, and (c) is one env-var typo (`EXECUTE=False`) away from ungated shell... gated only by an interrupt whose state lives in that same agent-writable history file. Each ADR signed off on its own slice; no document assesses the assembled system. Uvicorn's localhost default binding is the main mitigating factor, and it is an implicit default rather than a stated invariant — nothing in the scripts or settings pins the bind address.

Smaller items: `docker-compose.yml` ships `agentic:agentic` credentials with a comment saying "override via .env for anything real," but `02_dedup.sql`/`01_init.sql` and `config.py`'s `DATABASE_URL` default bake the dev credentials in as the assumed reality. The eval-optimizer fork path sets `LocalBackend(enable_execute=True)` headlessly — acknowledged in `fork_check.py`'s docstring ("run on a machine you trust"), which is the right disclosure attached to the wrong control (a docstring, not a gate).

## 7. What holds up well

For balance, and because a critique should also identify load-bearing strengths: the vendor hygiene design (pristine upstream + patch file + `VENDOR.txt` + `revendor_check.py` + `verify_core.py` as the consumer gate) is a genuinely disciplined pattern, and the patch itself is minimal, well-documented, and written to rebase cleanly. `agent-web`'s test suite tests real behavior over the real ASGI app with the real wire protocol — the interrupt tests in particular verify the exact property that matters (tool not executed before approval, executed exactly once after). The `main.py`/`app.py` split to keep import-time side effects out of tests, the env-leakage lessons encoded in `conftest.py`/`make_settings`, and the resilient-degrading MCP wrapper with a `/debug/mcp` explainability endpoint all reflect real operational scar tissue. The ADR corpus, staleness aside, is a far better decision record than most repositories of any size carry. These strengths are why the incongruities in §3 matter: the repo has earned the right to be held to its own documented standards, and §3 is precisely where it fails them.

## 8. Summary of findings by severity

**Critical (contract violations with runtime consequences):** checkpoint persistence claimed but per-request ephemeral; authoritative history writable by the agent it governs; host code-execution via always-on fork `test_command` bypassing the approval-gate story; self-graded fork selection metric; `EXECUTE=False`-style case-sensitivity enabling gated features.

**Major (correctness/robustness):** partial-pass merge threshold; silent judge fallback on any exception; no branch cancellation on timeout; concurrent-run history clobbering; thread-slug collisions; NVIDIA key required for non-NVIDIA runs with no OpenRouter key validation; agent/MCP toolset frozen at startup vs. live-reporting debug endpoint.

**Moderate (drift and hygiene):** stale NVIDIA/GLM-5.1 facts in four documents including agent-injected AGENTS.md; template-vs-default model mismatches; `Obsolete/` referenced but absent; vendor provenance unverifiable; no root LICENSE; no CI; 422 double-encoding; retry-coverage claim gaps; fork-config scatter; port 8000/8801 contradiction.

**Minor (residue):** four generations of dead code live in the package; scratchpad committed as docs; registry export, curl scratch file, and tsbuildinfo committed; aspirational catalogs referencing an undefined env root; three conflicting Python floors; personal absolute paths throughout; bypassable pip hook described as enforcement.
