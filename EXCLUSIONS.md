# EXCLUSIONS.md — the filtered-first-commit manifest (Phase 0.1a)

Harness2 is a fork of `github.com/Flux29/Harness @ 77f465e`. Git history is
permanent: anything present in commit 1 lives in the fork's history forever. So
known-contaminated artifacts were excluded **before** the first commit, not
deleted in a later phase. This file is the committed, auditable record of what
was excluded and why.

## Admission criteria (both must hold)

A path is excluded only if it is **both**:

- **(a) high-certainty inert** — zero import, runtime, test, or tooling
  reference anywhere in the tree, so exclusion cannot change any behavior; and
- **(b) a contamination or security risk** — machine state, personal data, raw
  prompts, or runtime output that should never enter a public fork's permanent
  history.

Whole-file contamination is what this manifest targets. Files that merely
*contain* a personal path on some lines (`verify_core.py`, READMEs, HANDOFF,
`schema.py`) are **edited** in Phase 1.5, not excluded — an edited line in
history is harmless.

## Deliberately NOT excluded

Despite being critique findings, these fail criterion (a) or belong to a later
decision:

- **Legacy eval-optimizer modules** (`graph.py`, `loop.py`, `validate.py`, dead
  schemas) are **test-load-bearing** — `test_smoke.py` imports `agents.Verdict`,
  `test_validate.py` imports `validate.parse_artifact`. Their fate is ADR 6.5's,
  not a commit-zero guess.
- **Files with personal paths** (`verify_core.py`, READMEs, HANDOFF, `schema.py`)
  need *edits* (Phase 1.5), not exclusion.
- **Everything else under `vendor/pydantic-deepagents`** stays byte-identical per
  the pristine-vendor rule.

## Baseline consequence

Because commit 1 is filtered, byte-parity between Harness2's first commit and
Harness1 is replaced by **this manifest plus a sha256 list of every retained
file** — see the appendix, [`EXCLUSIONS.sha256.txt`](EXCLUSIONS.sha256.txt)
(619 tracked files at fork time). All Phase 0.2 behavioral baselines are captured
from the original Harness repo, which continues to exist as the untouched
reference; nothing in the parity harness ever depended on the excluded files.

The appendix is the canonical Tier-1 (byte-equality) reference for Gate 1: any
PR that claims not to touch a file must leave its sha256 unchanged.

### Verifying the appendix

```sh
# from repo root; prints nothing and exits 0 if every retained file is intact
git ls-files | grep -v '^EXCLUSIONS' \
  | while IFS= read -r f; do sha256sum "$f"; done \
  | sed 's/ \*/  /' | diff - EXCLUSIONS.sha256.txt
```

> Note on the excluded targets: at fork time these paths are already absent from
> the tracked tree (the initial commit is the filtered tree). This manifest is
> the audit record of *why* they are absent, not a list of pending deletions.
